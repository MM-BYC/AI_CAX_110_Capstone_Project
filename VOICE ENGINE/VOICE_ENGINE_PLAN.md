# VOICE ENGINE Plan

## Mission

Build a standalone native Python voice engine from scratch for realtime audible speech-to-speech translation between two people on a phone call over WiFi.

The engine will:

- Receive live microphone audio from Person A and Person B.
- Detect each speaker's input language from configured call parameters.
- Translate speech into the other participant's target language.
- Generate audible translated speech in a cloned approximation of the original speaker's voice.
- Stream only engine-generated translated audio to the opposite participant.
- Never pass raw caller audio directly to the other participant.
- Prioritize the lowest practical latency.
- Prefer delay, confidence gating, or omission over hallucinated translated speech.

No OpenAI, no external LLM APIs, no hosted AI speech services, and no existing cloud AI tools will be used. The system will be built as a native Python engine using advanced Python audio, DSP, ML, and realtime networking libraries.

## Core Call Scenario

```text
Person A phone mic
  -> WiFi/WebRTC/RTP
  -> VOICE ENGINE
  -> A-to-B speech pipeline
  -> translated cloned speech audio
  -> Person B phone speaker

Person B phone mic
  -> WiFi/WebRTC/RTP
  -> VOICE ENGINE
  -> B-to-A speech pipeline
  -> translated cloned speech audio
  -> Person A phone speaker
```

Person A only hears translated output generated by the engine from Person B's speech.

Person B only hears translated output generated by the engine from Person A's speech.

Raw input audio is never played across the call.

## High-Level Architecture

```text
Phone Client A
  mic audio frames
    |
    v
Call Transport Layer
    |
    v
Audio Ingress A
    |
    v
Jitter Buffer
    |
    v
Denoise / Echo Suppression
    |
    v
Voice Activity Detection
    |
    v
Streaming ASR
    |
    v
Phrase Committer
    |
    v
Translation Engine
    |
    v
Terminology + Alignment Guard
    |
    v
Voice Clone TTS
    |
    v
Audio Egress to B
    |
    v
Phone Client B speaker
```

The same pipeline runs in reverse for B-to-A.

## Primary Engine Components

### 1. CallSessionManager

Owns the lifecycle of a translated call.

Responsibilities:

- Create a two-person call session.
- Store participant language settings.
- Store target output language settings.
- Route A input to B output.
- Route B input to A output.
- Track connection state, reconnects, mute, end call, and idle state.
- Prevent raw audio passthrough.
- Enforce call policy when both participants speak at the same time.

Example conceptual session shape:

```text
CallSession
  participant_a
    source_language: "Spanish"
    target_language_for_other_side: "English"
    voice_profile: A speaker embedding
  participant_b
    source_language: "English"
    target_language_for_other_side: "Spanish"
    voice_profile: B speaker embedding
  mode: translated_audio_only
```

### 2. Phone Transport Layer

Handles realtime phone audio over WiFi.

Recommended first transport:

- WebRTC for browser-based phone clients.
- Python `aiortc` for native Python WebRTC handling.

Alternative later:

- RTP over UDP for native mobile clients.
- QUIC/WebTransport if browser support and implementation quality are acceptable.

Transport requirements:

- 20 ms audio frames.
- Low jitter.
- Reconnect support.
- Client sends mic audio only.
- Client receives translated synthesized audio only.
- No direct peer-to-peer raw audio playback.

### 3. AudioIngress

Receives phone mic frames and converts them into normalized internal audio buffers.

Responsibilities:

- Decode inbound audio frames.
- Normalize sample rate, likely 16 kHz or 24 kHz.
- Maintain timestamps.
- Handle packet loss and jitter.
- Feed the VAD and ASR path continuously.

Candidate Python libraries:

- `aiortc`
- `av`
- `numpy`
- `scipy`
- `soundfile`
- `torchaudio`

### 4. JitterBuffer

Smooths network timing without adding excessive delay.

Requirements:

- Small adaptive buffer.
- Target 40-80 ms network tolerance.
- Drop or stretch frames under network instability.
- Emit stable frame cadence to downstream models.

### 5. Denoise and Echo Suppression

Required because phone speakers can leak translated engine output back into the mic.

Controls:

- Client-side acoustic echo cancellation.
- Server-side echo suppression.
- Optional output fingerprinting to detect engine audio re-entering the mic.
- Directional gating when output audio is playing loudly.
- Noise suppression before ASR.

Important policy:

```text
Engine output must not be re-captured and translated again.
```

Candidate libraries:

- `webrtcvad` for VAD.
- WebRTC audio processing bindings if suitable.
- `rnnoise` bindings if used locally.
- Custom spectral subtraction or learned denoiser later.

### 6. Voice Activity Detection

Separates speech from silence in realtime.

Requirements:

- 10-20 ms frame decisions.
- Low start latency.
- Conservative end-of-speech detection.
- Support barge-in and interruptions.
- Never wait for full sentences before starting downstream work.

Initial approach:

- `webrtcvad` for first implementation.
- Later replace or augment with a trained neural VAD if needed.

### 7. Streaming ASR Engine

Custom speech recognition model trained and served locally.

Goal:

```text
live speech audio -> partial source-language tokens with timestamps and confidence
```

Recommended architecture direction:

- Start with CTC model for a narrow first milestone.
- Move to Conformer/RNN-T style streaming ASR for better latency and accuracy.
- Export trained models to ONNX for production inference.

Outputs:

- Partial transcript tokens.
- Stable transcript tokens.
- Token timestamps.
- Token confidence.
- Segment confidence.

ASR must support:

- Source language parameter.
- Realtime partial output.
- Speaker-specific acoustic variation.
- Phone audio quality.
- Accents and noisy WiFi environments.

### 8. Phrase Committer

Converts unstable ASR token streams into safe translation chunks.

The engine must not translate every word independently because grammar, word order, and meaning differ across languages.

Use token states:

```text
unstable: may still change, never spoken aloud
pending: likely stable, can prepare translation/TTS internally
committed: safe to translate, synthesize, and stream
```

Strategy:

- Commit short stable phrases.
- Keep trailing uncertain tokens in pending state.
- Delay low-confidence fragments.
- Allow internal revision before audio is played.
- Never play speculative audio unless it is committed.

Example:

```text
ASR stream: "I want to go to..."

Committed: "I want"
Pending: "to go to"
Unstable: latest token
```

### 9. Translation Engine

Custom translation model trained locally.

Goal:

```text
source-language committed phrase -> target-language phrase
```

Requirements:

- Language description must be a parameter input.
- Support participant A source -> participant B target.
- Support participant B source -> participant A target.
- Preserve names, numbers, dates, currency, measurements, and proper nouns.
- Support domain-specific terminology.
- Produce short phrase-level translations with minimal latency.

Recommended architecture direction:

- First milestone: constrained phrase-based or small neural seq2seq translation for one language pair.
- Later: Transformer translation model trained on high-quality parallel corpora.
- Add alignment model for source-target consistency.
- Export to ONNX and quantize for production.

### 10. Terminology and Anti-Hallucination Guard

This is mandatory for safe audible translation.

The engine must prefer delay or silence over invented output.

Checks:

- ASR confidence threshold.
- Translation confidence threshold.
- Source-target alignment validation.
- Named entity preservation.
- Number/date/currency preservation.
- Required glossary terms.
- No unsupported extra entities.
- No unexplained expansion.
- No target speech if input confidence is too low.

Possible guard behavior:

```text
If ASR confidence is low:
  hold phrase and wait for more context.

If translation alignment fails:
  retry constrained decoding.

If named entities are missing:
  block audio output and regenerate.

If still uncertain:
  skip, delay, or emit an internal warning.
```

No hallucinated audible content should be sent to either caller.

### 10.1 Translation Memory and Confidence Learning

The realtime path should use a local translation-memory module before any
remote vector lookup. The package now has an internal persistent JSONL memory
path plus an in-process vector index. Pinecone can mirror approved examples for
external vector storage, but the live call must never wait on network retrieval
before speaking when a local match is available.

The local module stores approved source phrase to target phrase corrections,
glossary constraints, namespace/domain metadata, and lightweight grammar-profile
signals learned from reviewed examples. During a call, the orchestrator can use
exact or high-similarity approved matches immediately, treat weak matches as
confidence signals, and lower the score of translations that do not resemble
known safe structure for the language pair.

This does not prove that hallucination is impossible. It reduces the chance that
unsupported text reaches TTS by making the engine prefer known good phrases,
confidence-gated translation, and silence or delay when the candidate cannot be
justified. The same persisted examples become the offline dataset for later
fine-tuning of translation and grammar scoring models.

The runtime package also owns an automatic feedback model. After a candidate
translation is produced and before TTS runs, this model checks approved memory,
deterministic safety signals, and an optional host-provided correction provider.
When a higher-confidence correction is available, it replaces the candidate,
persists the corrected pair into translation memory, and appends the example to
the training dataset for offline model improvement.

The fast LLM reviewer and correction generator live inside VOICE ENGINE as
`GroqTranslationReviewer`. The app should not own this review loop; it only
configures credentials or model names. The orchestrator creates the reviewer per
room when `VOICE_ENGINE_GROQ_API_KEY` or `GROQ_API_KEY` is available.

The Groq translation agent also lives inside VOICE ENGINE as
`GroqTranslationEngine`. The app should not provide its backend translation
agent for room audio. The orchestrator creates the package translator per room
when credentials are available, then runs feedback and memory learning before
TTS.

The agent implementations live under `voice_engine/agents/`:

- `translation_agent.py`
- `quality_review_agent.py`

The engine and reviewer wrappers call these package-local agents. No backend app
agent is required for VOICE ENGINE feedback or correction.

### 11. Speaker Embedding Engine

Builds a compact voice identity representation from each participant's speech.

Modes:

- Pre-call enrollment: best quality.
- First-call adaptive enrollment: learn from first 10-30 seconds.
- Generic voice fallback until enough speech exists.

Requirements:

- Call-scoped voice profiles by default.
- Persistent voice profiles only with explicit user consent.
- Voice embeddings should be stored securely if persisted.
- Raw enrollment audio should never be sent to the other caller.

### 12. Voice Clone TTS

Generates target-language speech in an approximation of the source speaker's voice.

Recommended architecture direction:

- Speaker encoder.
- Text/phoneme frontend for target language.
- Acoustic model conditioned on speaker embedding.
- Neural vocoder.

Possible model family direction:

- FastSpeech2-style acoustic model for controllable low-latency generation.
- HiFi-GAN-style vocoder for fast waveform generation.
- Streaming vocoder later.

First target should be "same-speaker timbre approximation," not perfect cloning.

Output:

- Small audio chunks suitable for realtime streaming.
- Consistent sample rate.
- Stable pacing.
- Natural enough call quality.

### 13. AudioEgress

Streams generated translated audio to the opposite phone.

Responsibilities:

- Buffer synthesized audio just enough for smooth playback.
- Avoid excess delay.
- Handle packet loss.
- Duck, interrupt, or flush output when call state requires.
- Never mix raw source audio into output.

### 14. LatencyMonitor

Tracks every stage in the pipeline.

Metrics:

- Capture latency.
- Jitter buffer delay.
- VAD delay.
- ASR partial delay.
- Phrase commit delay.
- Translation delay.
- Guard delay.
- TTS generation delay.
- Egress/playback delay.
- Total A-to-B and B-to-A latency.

Latency must be visible from the first prototype.

## Latency Budget

Target practical budget:

```text
Network capture / jitter:       40-80 ms
VAD + denoise:                  20-50 ms
Streaming ASR partial:          150-300 ms
Phrase stabilization:           150-400 ms
Translation:                    50-150 ms
Voice clone TTS:                150-400 ms
Output jitter / playback:       40-100 ms

Realistic first production goal: 700 ms - 1.5 sec
Stretch goal:                   400-800 ms
```

Latency principles:

- Stream every stage.
- Never wait for full utterances unless required for correctness.
- Use committed phrase chunks, not full sentences.
- Keep models resident in memory.
- Use ONNX Runtime for production inference.
- Quantize models when quality allows.
- Use GPU where available.
- Profile every stage continuously.

## Simultaneous Speech and Barge-In

The engine must handle both callers talking at once.

Policy:

```text
If A speaks while B translated output is playing:
  continue capturing A.
  suppress echo.
  translate A independently.
  stream A's translated output to B.

If A and B speak simultaneously:
  run both pipelines independently.
  do not pass raw audio.
  avoid output feedback loops.
  optionally reduce output volume while local user speaks.
```

Do not make one direction block the other unless CPU/GPU limits require temporary throttling.

## Phone Client Requirements

The phone client is intentionally thin.

Client sends:

- Mic audio frames.
- Mute state.
- Call state.
- Participant language settings.
- Optional speaker enrollment permission.

Client receives:

- Translated synthesized audio only.
- Optional captions for debugging.
- Connection and latency state.

Client must:

- Use echo cancellation.
- Prefer headphones when possible.
- Avoid playing remote raw audio.
- Handle reconnects.
- Handle screen lock and WiFi drops gracefully.

## Data Requirements

This engine cannot become accurate without serious training data.

ASR data:

- Thousands of hours of transcribed speech per language.
- Phone-quality audio.
- Accents.
- Noise conditions.
- Meeting/call style speech.

Translation data:

- High-quality parallel corpora.
- Domain-specific phrase pairs.
- Terminology dictionaries.
- Human-reviewed bilingual examples.

TTS and voice cloning data:

- Multi-speaker speech corpus.
- Speaker-labeled recordings.
- Target-language phoneme coverage.
- Clean audio with alignments.
- Enrollment samples per user.

## Phased Build Plan

### Phase 1: Realtime Transport and Audio Harness

Goal:

```text
Two phones connect over WiFi.
Server receives both mic streams.
Server sends generated test audio back to opposite side.
No raw audio passthrough.
```

Deliverables:

- Call session lifecycle.
- A/B audio ingress.
- A/B translated-output-only egress placeholders.
- Jitter buffer.
- Latency logging.

### Phase 2: One-Way Speech Pipeline

Goal:

```text
Person A speaks.
Engine transcribes A.
Engine translates to B target language.
Person B hears synthetic translated audio.
```

Constraints:

- One language pair.
- Generic synthetic voice.
- Captions enabled for debugging.

### Phase 3: Bidirectional Translation

Goal:

```text
A -> translated audio to B
B -> translated audio to A
```

Deliverables:

- Two independent pipelines.
- Direction-specific language config.
- Simultaneous speech handling.
- Output queue management.

### Phase 4: Low-Latency Streaming

Goal:

Reduce delay by streaming each stage.

Deliverables:

- Streaming ASR partials.
- Phrase commit strategy.
- Streaming translation chunks.
- Streaming TTS chunks.
- Stage-by-stage latency dashboard.

### Phase 5: Anti-Hallucination Guard

Goal:

Prevent unreliable audible translations.

Deliverables:

- ASR confidence thresholds.
- Translation confidence thresholds.
- Named entity checks.
- Number/date/currency preservation.
- Glossary enforcement.
- Alignment validation.
- Block/delay behavior for uncertain output.

### Phase 6: Voice Clone

Goal:

Generate translated target-language speech in the source speaker's voice.

Deliverables:

- Speaker enrollment.
- Speaker embeddings.
- Speaker-conditioned TTS.
- Generic voice fallback.
- Secure call-scoped voice profile handling.

### Phase 7: Production Optimization

Goal:

Make the engine fast and stable enough for real calls.

Deliverables:

- ONNX export.
- Quantization.
- GPU inference path.
- CPU fallback path.
- Model warmup.
- Reconnect behavior.
- Error recovery.
- Load testing.

## First Milestone

The first milestone should not include full voice cloning.

Build this first:

```text
Two phones on WiFi.
Person A speaks English.
Person B hears Spanish synthetic speech.
Person B speaks Spanish.
Person A hears English synthetic speech.
No raw audio passthrough.
Captions available only for debugging.
```

Once this is stable and measurable, add voice cloning.

## Risks

Major risks:

- General ASR training requires large datasets.
- High-quality machine translation requires large bilingual corpora.
- Realtime voice cloning is computationally expensive.
- Perfect word-level translation is linguistically unreliable.
- Phone echo can create feedback loops.
- Ultra-low latency conflicts with translation correctness.
- CPU-only inference may not meet latency goals.

Mitigation:

- Start with one language pair.
- Start with narrow vocabulary/domain.
- Build measurable latency instrumentation first.
- Use phrase-level commit, not word-level audible output.
- Prefer confidence-gated delay over hallucination.
- Add voice clone only after the speech-translation-speech path is stable.

## Non-Negotiable Design Rules

- No raw caller audio is played to the opposite participant.
- No external LLM or hosted AI service is used.
- No hallucinated audible output should be emitted.
- Low-confidence phrases are delayed or blocked.
- Names, numbers, dates, and domain terms must be preserved.
- All heavy inference models must be loaded once and kept warm.
- Every pipeline stage must emit latency metrics.
- Voice cloning must use consent-controlled enrollment.
- The system must support bidirectional realtime calls.
- The implementation must remain modular so ASR, translation, and TTS can be improved independently.
